# SeedThought 2 コンセプト — 倉庫からアトリエへ

作成: 2026-07-08（Kotaroさんとの対話セッションから）

## 出発点（対話で確定した診断）

- 商用全振り（2026-07-06決定）の方向は**正しい**。求める快感は「発信の快感」。
- 惜しさの正体は「**新しいものを生み出す感じがない**」こと。
  - 情報は集約されているのに、情報×情報の掛け合わせが起きない。
  - 保存したポストが「実はすごい発見」でも、**なぜすごいのか**をアプリが教えてくれない。
  - 成功プロンプト等を保存しても、構造を学んでアレンジする、という**使い方**ができない。
  - 「積み上がらない」「触りたくならない」。開いても何も待っていない。
- SeedThought 1 の本質的な限界: **保存→分類→学習カード→下書き** という一直線のパイプラインで、1ポストずつ孤立して処理される。倉庫に出力の蛇口が付いているだけ。

## SeedThought 2 のコアループ

```
素材（ポスト/リサーチ/思考ダンプ）
  → 解読（なぜすごいか・何が変わったか）
  → 掛け合わせ（素材×素材、素材×型）
  → 創造（背景文脈つきの発信ネタ）
  → 資産化（型・示唆がライブラリに積み上がる）
  → 資産が次の掛け合わせの素材になる（複利ループ）
```

直線ではなくループ。フローだけでなくストックが育ち、ストックが次のフローを強くする。

## 3つの新しい器官 + 1つの土台

### 1. 解読器（Context Decoder）— enrich の再定義

保存アイテムに対して「カテゴリと要約」ではなく**意義**を解読する:

- **なぜすごいのか**（significance）
- **before/after**: 今まで◯◯しかできなかった → これが出た → こう変わる
- **誰にとって何が変わるか**

この before/after こそ「何も知らない人でも理解できる」商用アウトプットの核材料。
解読の出力がそのまま発信ドラフトの背景文脈として注入される（商用純化タスクの
「読者像注入」と接続）。

コスト設計: 通常解読（保存済みテキストのみ、既存Gemini）をデフォルト、
Web検索を伴う深掘り解読はオプトイン＋コスト明記（課金API回避の原則）。

### 2. 抽出器（Pattern Extractor）— 新規

成功例（画像プロンプト、構文、戦術）から**再利用可能な構造＝型**を抽出する。

例: 画像生成の成功プロンプト → 構成要素の解剖（主題/スタイル/制約/効いているトリック）
＋ なぜ効くのかの説明 → 型ライブラリに登録。

素材が「読むもの」から「使うもの」になる。

### 3. 掛け合わせエンジン（Synthesis Engine）— 新規、push + pull 両方

- **push（触りたくなる動機の設計）**: アプリを開くと「今日の掛け合わせ」が待っている。
  「先週保存したAと昨日のB、掛けるとこういうネタになる」という提案が向こうから来る。
- **pull**: 素材や型を自分で2〜3個選んで「これとこれで何ができる？」と指示する。

出力は解読済み文脈を背負った発信ネタ候補 → 既存の2パス生成（下書き→改稿）に流れる。

### 4. 資産庫（Asset Library）— 「積み上がる」の正体

型・示唆・掛け合わせの当たりが第一級のエンティティとして蓄積される。
資産は次の掛け合わせの素材になる（複利）。★知識マップはこの資産庫のビューとして実装する。

ゴール地点（確定）: **発信ネタになるまで** ＋ **再利用資産になるまで**。
アプリ内での実験実行（画像生成の実行など）はスコープ外。実験は外の実ツールでやる。

## 既存機能のマッピング

| 既存 | SeedThought 2 での位置づけ |
|---|---|
| X同期・ブックマークレット | 素材の入口（維持） |
| 学習カード生成（本体+補足の2並列） | 解読器を3本目の並列呼び出しとして統合（2026-07-08実装済み）。enrich はURLカード取得専用なので対象外 |
| 学習カード | 解読器に統合（2026-07-08決定）。本質（仕組みの理解・原典/系譜）は解読出力の「仕組みの解読」セクションとして生きる |
| 2パス下書き生成 | 創造の出口として維持。掛け合わせ結果を入力に受ける |
| リサーチ（quick/deep） | 素材源として維持。ネタキュー構想は掛け合わせpushに統合 |
| 知識マップ(/map) | 資産庫のビューとして再設計 |
| ホーム | 「今日の発見・今日の掛け合わせ」が待つ場所に再設計 |

## 実装順と状況

1. **解読器** ✅ 2026-07-08実装済み。実装の要点:
   - DBマイグレーションなし。decode は LearningCard.outputJson 内の `decode` オブジェクトに格納
   - `generateLearningCard` を本体/補足/解読の3並列に拡張（`src/lib/ai/provider.ts`）。
     解読は「生成→検品」の直列チェーンを並列枠内で実行。体感時間 = max(本体, 補足, 解読+検品)
   - 解読は best-effort（失敗してもカードは本体で成立）。型: `DecodeOutput`（`src/lib/ai/types.ts`）
   - プロンプトは引用ファースト+金標準few-shot+禁止則（`buildDecodePrompt` / `buildDecodeReviewPrompt`）
   - 出力ルートは card.outputJson を丸ごと素材として渡すため、decode は発信生成に自動で流れる（配線不要）
   - 学習ページに「なぜすごいのか（解読）」「仕組みの解読」「型として抽出」「発信ネタの種」を表示
   - 抽出器の入口（extractedPattern）と掛け合わせフック（synthesisTags）は decode 内に既に含まれる
2. **資産庫スキーマ**: 型・示唆のエンティティ設計（Prisma）。decode.extractedPattern を第一級レコードに昇格
3. **抽出器の独立強化**: 保存アイテム→型の抽出を単独でも実行可能に（過去カードの一括解読含む）
4. **掛け合わせ push**: 「今日の掛け合わせ」。synthesisTags と資産庫を突き合わせる。触りたくなる体験の完成はここ
